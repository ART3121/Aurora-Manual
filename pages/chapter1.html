1.1 What is Aurora IDE? 1.1 O que é a Aurora IDE? 1.1 Qu'est-ce qu'Aurora IDE ?

Aurora IDE is a graphical development environment for designing, compiling, simulating and visualizing SAPHO softcore processors. Developed and maintained by the NIPSCERN Laboratory at the Universidade Federal de Juiz de Fora (UFJF) in collaboration with CERN, Aurora serves as the primary interface between the designer and the SAPHO ecosystem, abstracting the full compilation pipeline from C± source code down to synthesizable Verilog HDL into a single integrated workflow.

O Aurora IDE é um ambiente de desenvolvimento gráfico para projeto, compilação, simulação e visualização de processadores softcore SAPHO. Desenvolvido e mantido pelo Laboratório NIPSCERN da Universidade Federal de Juiz de Fora (UFJF) em colaboração com o CERN, o Aurora serve como interface principal entre o projetista e o ecossistema SAPHO, abstraindo o pipeline completo de compilação do código-fonte C± até o Verilog HDL sintetizável em um único fluxo de trabalho integrado.

Aurora IDE est un environnement de développement graphique pour la conception, la compilation, la simulation et la visualisation de processeurs softcore SAPHO. Développé et maintenu par le Laboratoire NIPSCERN de l'Universidade Federal de Juiz de Fora (UFJF) en collaboration avec le CERN, Aurora sert d'interface principale entre le concepteur et l'écosystème SAPHO, abstrayant le pipeline de compilation complet du code source C± jusqu'au Verilog HDL synthétisable en un flux de travail intégré unique.

Rather than invoking compiler tools manually, the user interacts with Aurora through structured project management, a built-in code editor, and a Full Build command that orchestrates the entire toolchain automatically. Each compilation stage can also be triggered individually through its dedicated terminal. Simulation results are visualized directly within the environment through GTKWave integration, and the synthesized processor architecture can be inspected via the PRISM RTL Viewer.

Em vez de invocar as ferramentas de compilação manualmente, o usuário interage com o Aurora por meio de gerenciamento estruturado de projetos, um editor de código integrado e um comando Full Build que orquestra todo o toolchain automaticamente. Cada estágio de compilação também pode ser disparado individualmente pelo seu terminal dedicado. Os resultados de simulação são visualizados diretamente no ambiente pela integração com o GTKWave, e a arquitetura sintetizada do processador pode ser inspecionada pelo PRISM RTL Viewer.

Plutôt que d'invoquer manuellement les outils de compilation, l'utilisateur interagit avec Aurora via une gestion structurée de projets, un éditeur de code intégré et une commande Full Build qui orchestre automatiquement toute la chaîne d'outils. Chaque étape de compilation peut également être déclenchée individuellement via son terminal dédié. Les résultats de simulation sont visualisés directement dans l'environnement via l'intégration GTKWave, et l'architecture du processeur synthétisé peut être inspectée via le PRISM RTL Viewer.

Aurora is built with Electron and runs exclusively on Microsoft Windows. The current version covered by this manual is v4.27.10.

O Aurora é construído com Electron e roda exclusivamente no Microsoft Windows. A versão coberta por este manual é a v4.27.10.

1.2 The SAPHO Ecosystem 1.2 O Ecossistema SAPHO 1.2 L'écosystème SAPHO

1.2.1 What is SAPHO? 1.2.1 O que é o SAPHO? 1.2.1 Qu'est-ce que SAPHO ?

SAPHO (Scalable Architecture Processor for Hardware Optimization) is a configurable softcore processor developed at NIPSCERN. It is based on a Harvard architecture with a three-stage pipeline (Fetch → Decode → Execute), capable of executing one instruction per clock cycle. Unlike a fixed processor, SAPHO's architecture is synthesized at design time: the user describes the processor's behavior in C± and the toolchain generates only the hardware blocks required by that specific program, leaving unused ALU operations, memory, and logic out of the synthesis.

SAPHO (Scalable Architecture Processor for Hardware Optimization) é um processador softcore configurável desenvolvido no NIPSCERN. Baseia-se em uma arquitetura Harvard com pipeline de três estágios (Fetch → Decode → Execute), capaz de executar uma instrução por ciclo de clock. Ao contrário de um processador fixo, a arquitetura do SAPHO é sintetizada em tempo de projeto: o usuário descreve o comportamento do processador em C± e o toolchain gera apenas os blocos de hardware necessários para aquele programa específico, deixando operações de ULA, memória e lógica não utilizadas fora da síntese.

SAPHO (Scalable Architecture Processor for Hardware Optimization) est un processeur softcore configurable développé au NIPSCERN. Il repose sur une architecture Harvard avec un pipeline à trois étages (Fetch → Decode → Execute), capable d'exécuter une instruction par cycle d'horloge. Contrairement à un processeur fixe, l'architecture de SAPHO est synthétisée au moment de la conception : l'utilisateur décrit le comportement du processeur en C± et la chaîne d'outils génère uniquement les blocs matériels requis par ce programme spécifique, laissant les opérations ALU, la mémoire et la logique inutilisées hors de la synthèse.

This means every project in Aurora produces a physically distinct processor, customized for its specific application. The SAPHO architecture is particularly suited for signal processing and hardware acceleration tasks, as its arithmetic datapath can be configured for integer, floating-point, and native complex number operations.

Isso significa que cada projeto no Aurora produz um processador fisicamente distinto, customizado para sua aplicação específica. A arquitetura SAPHO é particularmente adequada para tarefas de processamento de sinais e aceleração de hardware, pois seu datapath aritmético pode ser configurado para operações com inteiros, ponto flutuante e números complexos nativos.

Cela signifie que chaque projet dans Aurora produit un processeur physiquement distinct, personnalisé pour son application spécifique. L'architecture SAPHO est particulièrement adaptée aux tâches de traitement du signal et d'accélération matérielle, car son chemin de données arithmétique peut être configuré pour des opérations sur des entiers, des nombres en virgule flottante et des nombres complexes natifs.

1.2.2 The C± Language 1.2.2 A Linguagem C± 1.2.2 Le langage C±

Processor behavior in SAPHO is described using C± (pronounced "C plus-minus"), a domain-specific language developed by NIPSCERN. It is important to understand that C± does not program a processor in the traditional sense: it describes the hardware behavior that will be synthesized. When you write a C± program, you are defining what circuits the processor will contain, not writing instructions to run on existing hardware.

O comportamento do processador no SAPHO é descrito usando C± (pronunciado "C mais-menos"), uma linguagem de domínio específico desenvolvida pelo NIPSCERN. É importante entender que o C± não programa um processador no sentido tradicional: ele descreve o comportamento do hardware que será sintetizado. Quando você escreve um programa C±, está definindo quais circuitos o processador vai conter, não escrevendo instruções para executar em hardware existente.

Le comportement du processeur dans SAPHO est décrit en C± (prononcé "C plus-moins"), un langage dédié développé par NIPSCERN. Il est important de comprendre que C± ne programme pas un processeur au sens traditionnel: il décrit le comportement matériel qui sera synthétisé. Lorsque vous écrivez un programme C±, vous définissez quels circuits le processeur contiendra, et non des instructions à exécuter sur du matériel existant.

C± is syntactically close to a subset of standard C, making it approachable for students with prior programming experience. Beyond the standard int and float types, C± introduces a native comp type for complex number operations, a key differentiator of the SAPHO architecture that enables hardware-level arithmetic on complex numbers without any software overhead. The language also provides a standard library of hardware-synthesized functions including abs(), fase(), sqrt(), sin(), cos(), and others.

O C± é sintaticamente próximo de um subconjunto do C padrão, tornando-o acessível para estudantes com experiência prévia em programação. Além dos tipos padrão int e float, o C± introduz o tipo nativo comp para operações com números complexos, um diferencial chave da arquitetura SAPHO que permite aritmética com complexos em nível de hardware sem nenhum overhead de software. A linguagem também fornece uma biblioteca padrão de funções sintetizadas em hardware, incluindo abs(), fase(), sqrt(), sin(), cos() e outras.

C± est syntaxiquement proche d'un sous-ensemble du C standard, le rendant accessible aux étudiants ayant une expérience préalable en programmation. Au-delà des types standard int et float, C± introduit un type natif comp pour les opérations sur les nombres complexes, différenciateur clé de l'architecture SAPHO permettant l'arithmétique sur les complexes au niveau matériel sans aucune surcharge logicielle. Le langage fournit également une bibliothèque standard de fonctions synthétisées en matériel, dont abs(), fase(), sqrt(), sin(), cos() et d'autres.

/* Processor configuration directives (generated by Processor Hub) */
#PRNAME procTest_00
#NUBITS 5
#NDSTAC 5
#SDEPTH 5
#NUIOIN 1
#NUIOOU 1
#NBMANT 2
#NBEXPO 2

/* Program entry point */
void main() {
    int a = 1;
    int b = 2;
    int c = a + b;
    out(0, c);   /* send value of c to output port 0 */
}

The block of #DIRECTIVE lines at the top of the file encodes the processor's architectural parameters. These are automatically generated by the Processor Hub (see Chapter 4) and it is recommended to keep them in sync with the processor configuration rather than editing them directly. The void main() function is the program's entry point and contains the user's logic.

O bloco de linhas #DIRETIVA no topo do arquivo codifica os parâmetros arquiteturais do processador. Esses valores são gerados automaticamente pelo Processor Hub (ver Capítulo 4) e recomenda-se mantê-los sincronizados com a configuração do processador em vez de editá-los diretamente. A função void main() é o ponto de entrada do programa e contém a lógica do usuário.

Le bloc de lignes #DIRECTIVE en haut du fichier encode les paramètres architecturaux du processeur. 1.4 Project Structure 1.4 Estrutura do Projeto 1.4 Structure du projet

Every Aurora project is organized into a fixed folder hierarchy that mirrors the stages of the compilation pipeline. When a new project is created, Aurora generates this structure automatically:

Todo projeto Aurora é organizado em uma hierarquia de pastas fixa que espelha os estágios do pipeline de compilação. Quando um novo projeto é criado, o Aurora gera essa estrutura automaticamente:

Chaque projet Aurora est organisé en une hiérarchie de dossiers fixe qui reflète les étapes du pipeline de compilation. Lors de la création d'un nouveau projet, Aurora génère automatiquement cette structure : 

ProjectName/
└── processorName/
    ├── Hardware/       ← Generated Verilog HDL files (core.v, ula.v, …)
    ├── Simulation/     ← Testbench files (procName_tb.v) and .vcd waveforms
    └── Software/       ← User-written source files
        ├── procName.cmm  ← C± source code (edit this)
        └── procName.asm  ← Assembly output (generated, do not edit)

The Software/ folder is where active development takes place. The .cmm file is the only file users are expected to edit directly. Files in Hardware/ and Simulation/ are fully managed by the toolchain and are regenerated on every Full Build.

A pasta Software/ é onde o desenvolvimento ativo ocorre. O arquivo .cmm é o único que os usuários devem editar diretamente. Os arquivos em Hardware/ e Simulation/ são totalmente gerenciados pelo toolchain e são regenerados a cada Full Build.

Le dossier Software/ est l'espace de développement actif. Le fichier .cmm est le seul fichier que les utilisateurs sont censés modifier directement. Les fichiers dans Hardware/ et Simulation/ sont entièrement gérés par la chaîne d'outils et sont régénérés à chaque Full Build.
